Introduction

Understanding PurpleForge

What problem does it solve?

In a SOC team, the most common question after a red team exercise is simple — and daunting:

"We know what the red team did. But would we have caught anything if it was a real attacker?"

Without a dedicated tool, the answer to this question gets lost in scattered Excel files, Notepad notes, emails, and memories. PurpleForge exists to answer it clearly, at any time.

❌ Without PurpleForge

  • Red team report as PDF — read once, filed away
  • No tracking of undetected techniques
  • Blue team doesn't know what to build first
  • CISO cannot measure progress over time
  • Each exercise starts from scratch
  • Sigma rules must be found manually

✅ With PurpleForge

  • Every technique logged with detection status
  • Prioritized detection backlog, automatically
  • Sigma rules suggested per technique in one click
  • Coverage measured as %, per tactic
  • Full history of exercises, comparable
  • ATT&CK Navigator export for leadership

Purple teaming explained simply

Purple teaming is a collaborative security practice that brings red (attackers) and blue (defenders) teams together — rather than pitting them against each other.

TeamRoleIn PurpleForge
🔴 Red teamSimulates the attackerDocuments techniques used, tools, attack outcomes
🔵 Blue teamDefends and detectsQualifies each technique, notes detection rules deployed
🟣 Purple teamCoordinatesTracks overall coverage, drives remediation

MITRE ATT&CK in 5 minutes

MITRE ATT&CK is a globally accessible knowledge base of adversary tactics and techniques, organized into 14 tactics (the stages of an attack), each containing dozens of specific techniques.

🗂️

The 14 ATT&CK Enterprise Tactics

Reconnaissance · Resource Development · Initial Access · Execution · Persistence · Privilege Escalation · Defense Evasion · Credential Access · Discovery · Lateral Movement · Collection · Command & Control · Exfiltration · Impact

Every technique has a unique identifier (e.g., T1003.001 for LSASS Memory Dump). PurpleForge uses this framework as a common language between red and blue teams.

Sigma detection rules

Sigma is an open, generic signature format for SIEM systems. A Sigma rule describes what to look for in logs to detect an attack technique. It can then be converted for any SIEM (Splunk, Wazuh, Elastic…).

💡

Good news

PurpleForge automatically downloads 2,900+ Sigma rules from the SigmaHQ community repository. For each ATT&CK technique in your campaign, it suggests matching rules — ready to copy into your SIEM.

Option A — Local installation (Windows)

1

Download Python 3.12

Go to python.org/downloads and install Python 3.12+. Check "Add Python to PATH" during installation.

2

Clone the repository

PS> git clone https://github.com/Yeni-raamia/purpleforge-compagnon.git
PS> cd purpleforge-compagnon
3

Create the virtual environment

PS> python -m venv .venv
PS> .\.venv\Scripts\Activate.ps1

⚠️ "Scripts disabled" error?

Run first: Set-ExecutionPolicy -Scope CurrentUser RemoteSigned

4

Install dependencies

(.venv) PS> pip install -r requirements.txt
5

Start the server

(.venv) PS> uvicorn app.main:app --host 0.0.0.0 --port 8080 --reload

Open http://localhost:8080 in your browser.

ℹ️ First launch

The app automatically downloads ATT&CK data (~40 MB) and Sigma rules (~30 MB). This takes 1–3 minutes and happens only once. Subsequent starts are instant.

6

Initial login

Default credentials: admin / admin. Change the password immediately via the Account menu.

Option B — macOS / Linux

# Clone and enter the directory
$ git clone https://github.com/Yeni-raamia/purpleforge-compagnon.git && cd purpleforge-compagnon

# Create and activate the virtual environment
$ python3 -m venv .venv && source .venv/bin/activate

# Install and launch
(.venv) $ pip install -r requirements.txt
(.venv) $ uvicorn app.main:app --host 0.0.0.0 --port 8080 --reload

Option C — Docker (recommended for production)

1

Install Docker Desktop

Download from docker.com/products/docker-desktop and launch the application.

2

Launch with one command

$ docker compose up

PurpleForge is available at http://localhost:8080. The database and ATT&CK data persist across restarts via Docker volumes.

3

Stop / Update

$ docker compose down                      # stop
$ git pull && docker compose up --build   # update

Verify the installation

# Run the automated test suite (64 tests)
$ python -m pytest tests/ -v
64 passed in 0.75s ✓
🎉

Installation successful!

If you see "64 passed", everything is working correctly. PurpleForge is ready to use.

3.1 — Creating your first campaign

1

Open the Campaigns page

Click Campaigns in the navigation. The "Tactical Ops Center" page opens — dark background, global stats at the top.

2

Click "+ New Campaign"

The circular button in the top right opens a side drawer panel.

3

Fill in the form

  • Name: e.g., "Red Team Q3 2026 — Lateral Movement"
  • Description: exercise context, tested scope
  • Tags: e.g., APT29, Ransomware, Internal
4

Click "Create"

The campaign appears in the grid. Click on it to access its detail page.

🚀

Tip: APT Templates

Instead of starting from scratch, use the pre-built templates: APT Templates → choose a threat group (APT28, APT29, Lazarus, FIN7…). The campaign is created with that group's characteristic techniques already loaded.

3.2 — Adding techniques from the matrix

1

Open the ATT&CK matrix

From the campaign page, click Open ATT&CK Matrix. The full matrix is displayed: 14 tactics as tabs, techniques listed below.

2

Browse by tactic

Click a tactic tab (e.g., "Credential Access") to see its techniques. Each shows its ATT&CK identifier and name.

3

Add a technique

Click + Add under each technique played during the exercise. It appears in the campaign instantly (HTMX, no page reload).

3.3 — Qualifying a detection

This is the core of PurpleForge. For each technique, the blue team must answer: "Would we have detected it?"

StatusMeaningNext action
✗ Not DetectedNo alert triggered during exerciseBuild a detection rule (high priority)
⚙ To BuildRule exists but not yet deployedDeploy and validate in SIEM
✓ DetectedAlert triggered and validatedDocument and move on
1

Open the technique card

On the campaign detail page, click a technique to expand its card.

2

Change the status

Select the status from the dropdown. The update is instant (HTMX) — the coverage bar at the top recalculates immediately.

3

Add notes

  • Red Team note: tool used, payload, access vector, attack outcome
  • Blue Team note: deployed rule, SIEM ticket, technical comment (Markdown supported)

3.4 — Sigma rule suggestions

1

Click "⚡ Sigma Rules"

On any technique card, this yellow button displays matching Sigma rules from SigmaHQ.

2

Read the suggestions

Each rule shows: title, required log sources (e.g., "Sysmon Event ID 10"), and the full YAML content.

3

Copy to your SIEM

Copy the YAML and import it into Wazuh, Elastic, Splunk, or any Sigma-compatible SIEM. Log the reference in the Blue Team note.

Tutorials

Step-by-step Tutorials

Tutorial 1 — First post-compromise exercise

📋 Context

Your red team just completed a 2-day exercise simulating an attacker with existing access to a Windows workstation. Techniques used: credential extraction (Mimikatz), lateral movement (PsExec), persistence (scheduled task).

1

Create the campaign

Campaigns → New Campaign → Name: "Post-Compromise Exercise — July 2026", Tags: internal, Windows

2

Add techniques from the matrix

ATT&CK Matrix → Add:

  • T1003.001 — LSASS Memory Dump (Credential Access)
  • T1021.002 — SMB/Windows Admin Shares (Lateral Movement)
  • T1053.005 — Scheduled Task/Job (Persistence)
3

Qualify each technique

  • T1003.001 → Not Detected — no Wazuh alert on LSASS access
  • T1021.002 → To Build — Sigma rule identified, not yet deployed
  • T1053.005 → Detected — alert raised via Sysmon Event ID 11
4

Get Sigma rules for T1003.001

Click ⚡ Sigma Rules → PurpleForge suggests "Credential Dumping via Procdump" and "LSASS Access via Task Manager". Copy the YAML and create the rule in Wazuh.

5

Create remediation tasks

For T1003.001 and T1021.002 → Edit → assign to a team member, set a deadline 2 weeks out → the remediation board updates automatically.

6

Export for leadership

⬇ Export Navigator → JSON opens on attack-navigator → colored matrix showing your 3 techniques. Perfect for the debrief report.

Tutorial 2 — Using an APT28 template

📋 Context

You want to assess your detection coverage against techniques characteristic of APT28 (Fancy Bear), a Russian GRU-linked group known for targeting government institutions.

1

Load the template

Campaigns → APT Templates → select "APT28 / Fancy Bear" → Create from this template. A campaign is created with ~15 characteristic techniques pre-loaded.

2

Review the techniques

The campaign contains spearphishing, credential theft, lateral movement techniques. Qualify each based on your current detection capabilities.

3

Honest assessment without a real exercise

Even without a red team exercise, you can qualify techniques based on your SIEM knowledge: "Do I have a rule for this?" This is a useful posture assessment.

4

Review the coverage page

Coverage → the stacked bar shows your detection rate against APT28 by tactic. If Credential Access is at 0% — that's an immediate priority.

Tutorial 3 — Managing remediation over 3 months

📋 Context

After several exercises, you have a backlog of 20 undetected techniques. You need to organize your blue team's work over the quarter.

1

Global Dashboard → "Hot spots"

The dashboard identifies techniques that appear across multiple campaigns. T1055 Process Injection on 3 campaigns = top absolute priority.

2

Assign and schedule

Global Remediation → for each technique: assign an owner, set a deadline. The kanban board visualizes In Progress / Blocked / Done.

3

Weekly check-in

Statistics → "Detection Rate Evolution" shows month-by-month progress. Share the chart in your weekly review meetings.

4

Run a validation exercise

After deploying rules, create a new campaign with the same techniques → previously "Not Detected" techniques should now be "Detected". This is concrete proof of progress.

Best Practices & Glossary

Best practices

🏷️ Consistent campaign naming

Adopt a standard format from the start:

[Type] [Year-Quarter] — [Theme]
Example: Red Team Q3-2026 — LockBit Ransomware
Example: Purple Q4-2026 — Initial Access

📅 Recommended cadence

FrequencyAction
After each exerciseCreate campaign, add and qualify all techniques within 48 hours
WeeklyUpdate remediation statuses, check deadlines
MonthlyReview global dashboard, adjust priorities
QuarterlyCompare two campaigns (Compare feature), present evolution

🎯 Using the Compare view

The Compare function (Campaigns → Compare) displays two campaigns side by side and highlights techniques that changed status. It's your most powerful progress indicator.

📊 Sharing with leadership

The ATT&CK Navigator JSON export is your best ally. Imported on attack-navigator.mitre.org, it produces a colored heatmap (red = undetected, green = detected) that anyone can understand — no technical jargon required.

💡

Practical tip

Export Navigator before AND after each remediation phase. The visual difference between the two matrices is the strongest argument for demonstrating your SOC team's value.
